# Design: 合併請款工作台（統一請款入口）
spec-id: 001-merged-payment-workbench

## 架構決策

### 決策 1: 不新增資料表
- **選擇**：利用 `quotation_items` 既有的 `merge_group_id`、`is_merge_leader`、`merge_color` 欄位，不建立獨立的 `merge_groups` 表
- **原因**：
  - Schema 已有完整的合併欄位，新增表格是過度工程
  - 團進團出保證組內狀態一致，合併組的狀態可由項目狀態推導
  - 減少 migration 複雜度和跨表 join
- **替代方案**：新增 `merge_groups` 表儲存組元資料（創建者、備註、狀態）→ 增加維護成本，且組狀態會與項目狀態冗餘

### 決策 2: 統一請款入口（D2）
- **選擇**：工作台為唯一的請款送出和審核入口
- **原因**：
  - 消除「從哪裡送的」追蹤問題
  - 報價單 DataGrid 職責單純化（只管成本資訊）
  - 審核者只需去一個地方
- **替代方案**：工作台只管合併，個別請款保留在原位（D1）→ 雙軌制容易混亂

### 決策 3: RPC 層級的團進團出
- **選擇**：合併組的送出/核准/駁回由專用 RPC 在 DB transaction 內處理
- **原因**：
  - 避免前端逐筆呼叫造成的部分成功問題
  - DB 層保證原子性（全成功或全失敗）
  - 減少網路往返
- **替代方案**：前端 loop 逐筆呼叫既有 RPC → 網路延遲大、可能部分失敗

### 決策 4: 工作台 Tab 分割
- **選擇**：按請款狀態分 Tab（待處理 / 審核中 / 被駁回），而非按來源分（專案 / 個人）
- **原因**：
  - 使用者的心智模型是「我現在要做什麼」（送出？審核？修正？）
  - 專案和個人請款可以混在同一個 Tab 內，用視覺區分
  - 審核者在「審核中」Tab 一次看到所有待審項目
- **替代方案**：按來源分 Tab → 審核者需要切換兩個 Tab 才能審完

### 決策 5: 合併組色帶複用既有調色盤
- **選擇**：複用 `MERGE_COLORS` 6 色調色盤
- **原因**：已驗證的配色、與既有 UI 一致
- **替代方案**：擴展色彩數量 → 6 色已足夠，超過反而難以區分

### 合併操作流程

```
使用者在「待處理」Tab 勾選多筆同帳戶項目
  ↓
點擊「合併選取項目」
  ↓
┌─ 驗證 ─────────────────────────────────────────────┐
│ 1. 全部同一 remittance_name？     ── 否 → 阻擋     │
│ 2. 全部同一 bank_info（JSON）？    ── 否 → 阻擋     │
│ 3. 全部 pending 狀態？             ── 否 → 阻擋     │
│ 4. 全部未加入其他合併組？           ── 否 → 阻擋     │
│ 5. 預計請款月份是否一致？           ── 否 → 跳警告   │
└────────────────────────────────────────────────────┘
  ↓ 驗證通過
MergeConfirmDialog 開啟
  - 列出所有待合併項目
  - 使用者選擇主項（預設第一筆）
  - 確認
  ↓
RPC: create_quotation_merge_group(item_ids, leader_id)
  ↓
UI 更新：合併組以色帶卡片呈現
```

## v1.1 新增設計決策

### 決策 6: 工作台行內上傳範圍
- **選擇**：只支援「發票號碼 + 附件上傳」，不支援成本金額等資料面欄位
- **原因**：工作台定位是「請款流程終端」，資料面編輯留在報價單 DataGrid
- **替代方案**：全欄位編輯 → UI 過於複雜，與 DataGrid 功能重疊

### 決策 7: 依帳戶類型分組
- **選擇**：依 `bankType` 分三區（勞報/公司行號/未填寫），以對應戶名為分組標題
- **原因**：
  - 完全對齊下游匯款邏輯（已確認清單的 `isCompanyAccount` 分流）
  - 使用者一眼看出「個人匯款」和「公司匯款」兩大類
  - 「未填寫資料」區塊提醒補齊，避免送出後匯款時才發現缺資訊
- **替代方案**：用 `remittance_name` 分組（v1.0）→ 空值多、不反映帳戶類型

### 決策 8: 合併 dialog 增加月份步驟
- **選擇**：選主項 → 選月份 → 確認，三步合併
- **原因**：讓「已確認請款清單」能按月份正確歸類，解決大量「未指定月份」問題
- **UI 規則**：組內月份若已一致 → 自動帶入；全部為空或不一致 → 必須手動選

### v1.1 分組資料流

```
kols.bank_info (JSONB)
  ├─ bankType = 'individual'
  │   └─ personalAccountName → 「勞報（個人戶）」區塊標題
  ├─ bankType = 'company'
  │   └─ companyAccountName → 「公司行號」區塊標題
  └─ bankType = null / 戶名為空
      └─ 「未填寫資料」區塊（提醒補齊）

get_workbench_items() 已回傳 kol_bank_info，前端 parse 即可。
```

### v1.1 進項管理合併標示

```
accounting_expenses 查詢：
  .select('*, payment_requests(...), quotation_items(...)')
           ↓                           ↓
```

## 依賴關係

```
新元件依賴：
  useWorkbenchItems     → supabase.rpc('get_workbench_items')
  useWorkbenchMerge     → supabase.rpc('create/dissolve_quotation_merge_group')
  useWorkbenchSubmission → supabase.rpc('submit/withdraw_merge_group/single_item')
  useWorkbenchReview    → supabase.rpc('approve/reject_merge_group')
                        → 既有 approve_quotation_item() 邏輯（被 approve_merge_group 內部呼叫）

修改的既有元件：
  QuotationItemsList    ← 移除按鈕、新增 MergeBadge
  expense-claims/page   ← 移除送出/審核按鈕
  Sidebar               ← 新增導覽項
  confirmed-payments    ← 新增合併標示

快取失效鏈：
  合併/送出/撤回/核准/駁回 → invalidate ['workbenchItems']
  核准 → 額外 invalidate ['paymentConfirmations', 'accountingExpenses']
  報價單 DataGrid 修改成本 → invalidate ['workbenchItems']（成本更新反映到工作台）
```
