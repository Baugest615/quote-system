# 討論：匯款日期架構 — 分日顯示 + 審核階段填入 + 同步機制
日期：2026-03-07
狀態：進行中

## 背景

目前「已確認請款清單」以**月份**為單位合併同一匯款對象的款項。當同一個人戶在同月有不同天匯款的需求時，金額會被統整在一起，造成財務人員難以區分每次實際匯款金額。

此外，匯款日期目前是在「已確認請款清單」才填入，但理想流程應該是在請款工作台「審核通過」時就決定匯款日期，並自動同步到下游（確認清單、進項表）。

### 現況資料流
```
請款工作台（審核中）→ 核准 → 已確認請款清單 → 設定匯款日期 → 同步進項表
                      ↑                        ↑
                   無日期欄位              此處才填日期
```

### 目標資料流
```
請款工作台（審核中）→ 填入匯款日期 → 核准 → 已確認請款清單（按匯款日分組）→ 進項表（自動同步）
                      ↑                         ↑
                   新增日期欄位              不同日期分開顯示
```

## 討論紀錄

### Q1: 不同匯款日的同一匯款對象，是否需要在已確認請款清單中分開顯示？

**結論：是，按匯款日分開。**

理由：
- 已確認請款清單的核心用途是「產出匯款指示」，不同天就是不同筆銀行交易
- 合併顯示會迫使財務人員自行拆分，增加出錯風險
- 分開後每組金額 = 當天實際匯款金額，對帳更直觀
- 代扣門檻判斷也需要按日計算（所得稅法 §88），分組邏輯已存在

實作方向：在 groupKey 加入匯款日，例如 `acct_12345_2026-03-05`。未填匯款日的維持原本邏輯。

### Q2: 匯款日期應該在哪個階段填入？

**選項 A**：維持現況 — 在「已確認請款清單」填入
- 優點：不改工作台流程，改動最小
- 缺點：審核通過 ≠ 決定匯款日，兩步驟分離容易遺漏

**選項 B**：在請款工作台「審核通過」時填入
- 優點：審核 = 決定何時付款，語意更合理；一步完成
- 缺點：審核者需要同時決定匯款日，增加審核負擔；若匯款日之後需變更，需要回到工作台嗎？

**選項 C**：工作台審核時填入（預設值），已確認請款清單可覆寫
- 優點：兩邊都能操作，最大彈性
- 缺點：兩個入口可能造成資料不一致風險

**結論：選擇 C — 審核時填入預設值，確認清單可覆寫。**

理由：
1. 審核者通常知道這筆何時要付，填在審核動作中最自然
2. 但實際匯款日可能因銀行假日、資金調度等因素調整，確認清單保留修改權
3. 已確認請款清單上的修改為最終值（覆寫審核時的預設），下游同步以此為準
4. 現有 `remittance_settings.paymentDate` 機制不變，只是上游多一個預設來源

### Q3: 匯款日期如何存儲和同步？

**現有儲存架構**：
| 表 | 欄位 | 用途 |
|---|---|---|
| payment_confirmations | remittance_settings (JSONB) → paymentDate | 確認清單群組的匯款日期 |
| accounting_expenses | payment_date (date) | 進項表匯款日 |
| accounting_payroll | payment_date (date) | 薪資匯款日 |

**新增欄位**：
| 表 | 新增欄位 | 用途 |
|---|---|---|
| payment_requests | payment_date (date, nullable) | 審核通過時填入的預計匯款日 |

**同步流程**：
```
1. 工作台審核通過 → 寫入 payment_requests.payment_date
2. 確認清單載入時 → 讀取 payment_requests.payment_date 作為預設值
   （若 remittance_settings.paymentDate 已有值則以後者為準）
3. 確認清單修改日期 → 更新 remittance_settings.paymentDate
4. RPC update_remittance_settings() → 同步 accounting_expenses.payment_date
5. 薪資獨立更新 → accounting_payroll.payment_date
```

### Q4: 合併請款（同匯款對象多筆請款）如何處理匯款日？

**情境**：同一個 KOL 有 3 筆請款被合併到同一確認單

**選項 A**：合併時要求所有請款使用同一匯款日
- 優點：簡單，一組一個日期
- 缺點：不同專案可能有不同付款期限

**選項 B**：每筆請款各自保留匯款日，確認清單按日期再分組
- 優點：最精確
- 缺點：同一人可能被拆成多組

**結論：選擇 B — 各自保留，按日期分組。**

理由：
1. 這正是 Q1 的結論 — 不同匯款日就分開顯示
2. 同一人在確認清單中按匯款日出現多次，每組代表一筆銀行交易
3. 代扣門檻已按日獨立計算，分組與代扣邏輯一致
4. 若所有請款匯款日相同，自然合併為一組，行為不變

### Q5: 已確認請款清單的匯款日期還能修改嗎？

**結論：可以修改，且修改後自動同步下游。**

- 確認清單的匯款日期是最終值
- 修改後：RPC 同步 accounting_expenses.payment_date（現有機制）
- 若修改導致分組變化（原本同日→拆成不同日），UI 重新渲染即可反映

## 收斂結論

1. **已確認請款清單按匯款日分組**：同一匯款對象、不同匯款日 → 分開為不同群組
2. **匯款日在審核通過時填入**：新增 `payment_requests.payment_date` 欄位，審核 UI 加入日期選擇器
3. **確認清單可覆寫**：最終匯款日以 `remittance_settings.paymentDate` 為準
4. **進項表自動同步**：現有 RPC 機制不變
5. **合併請款按日期再分組**：不同匯款日的請款即使同一匯款對象也分開顯示

## 影響範圍評估

| 層級 | 影響 |
|------|------|
| DB | 新增 `payment_requests.payment_date` 欄位 |
| RPC | `update_remittance_settings()` 不需改（已支援按戶名批量更新） |
| 工作台 UI | ReviewSection 新增匯款日期選擇器 + 核准 API 帶入日期 |
| 確認清單 aggregation | groupKey 加入 paymentDate 維度 |
| 確認清單 UI | 初始化時讀取 payment_requests.payment_date 作為預設值 |
| 進項表 | 不需改（現有同步機制） |

**變更等級：Level 3（跨 DB + 工作台 + 確認清單 + aggregation）**
